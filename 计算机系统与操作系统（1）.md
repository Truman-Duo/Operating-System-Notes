# 计算机系统与操作系统（1）

## 第一章：计算机系统漫游

### 引言

自工业革命以来，机械化、自动化的生产生活方式深刻改变了人类社会。优秀的工具可以作为人的延伸提高效率，提高生活品质。在短短的百年间，人类就完成了对四肢的延伸，但对脑的思维——对智能的延伸却几经辗转，直到现在还在持续探索之中。

计算机就是其中最接近成功的一个尝试，它并不全面的延伸了人的智能，仅能代替人脑解决”可计算（见后）”的问题，但作为一个工具、机器，它也深刻影响了整个社会。

### 计算机的发展历程

1936 年，英国人图灵提出了一个理论模型系统——后世称为”图灵机”，该系统能在相当程度上完成只有人脑才可以解决的问题，这些问题统称为”可计算”问题。

早期的计算机是面向特定任务设计的专用机器。不同的问题需要不同的解决方案，但随着任务逐渐复杂，这种”特事特办”出现了问题：每更换一个任务，就需要重新制造机器，提升的效率不足以支持极高的成本。因此一个自然的想法是：

是否可以设计一种通用机器，只需略作调整——如人之间通过语言沟通调整行为，计算机层面称为调整”程序指令”，而不必全盘更换就能完成不同任务？

1944 年，冯·诺依曼在原有计算机的基础上优化了结构，实现了这个想法，后世把这样的结构布置称为**冯·诺依曼架构**。该架构的工作流程可抽象为三步：接受信息、处理信息、输出结果。也叫 **IPO 流程**。

其中”接受信息”与”输出结果”这两步本质上无异，都是计算机与外界环境的交互，统称为 **I/O（Input/Output，输入/输出）**。”处理信息”这一步是计算机的核心功能，由专门负责执行指令与计算的硬件单元完成，称为**处理器**。计算机还需要存储一些过程信息和中间结果，这需要有地方临时存放，这个地方称为**内存（Memory）**。

### 计算机硬件

根据冯·诺依曼架构，计算机至少需要包含以下几个硬件：

- 处理器（现代已经集成化为中央处理器）
- 内存
- 负责 I/O 的相关设备

中央处理器的结构与工作原理已在计算机组成原理讲义中说明，此处不再赘述。我们重点考察内存和 I/O 设备。

#### 内存

冯·诺依曼架构要求处理器仅能从内存中读取数据，我们知道，寄存器由触发器（详见计算机组成原理讲义）组合而成，工作速度与所在模块的处理单元匹配，但造价高、体积大，因此计算机一般不直接使用寄存器作为完整的内存。为节省成本，计算机的采用了多层存储结构，仅当与 CPU 相连的内存部分使用了寄存器，总容量只有数百字节，远不足以完整存放当今的程序指令。剩下的内存需求由 RAM ——一种基于另一种物理方法构建的存储结构代替，而且再进一步，这部分里也只有一小部分的速度快，称为 **Cache**，剩下的速度较慢，我们也把这部分最慢的称为**主存**。

```
从上到下速度依次变慢，造价变低：

寄存器（Register）
│   CPU 内部，数十个，每个 64 位，触发器电路构成
│
RAM ─┬─ SRAM：构成缓存（Cache）
     └─ DRAM：构成主存（Main Memory）
```

由于本身的物理构架，寄存器、Cache、主存断电数据都会立即丢失，这称为**易失性（Volatile）**。

注：后文提到”内存”时，通常特指主存。

#### 外存

为了长期存储大量程序和数据，计算机必须有能在断电后不丢失数据的存储结构——即**非易失性的存储 （NVS, Non-Volatile Storage**），这样的存储的速度不必要求与内存相当，使用时只需将数据拷贝到内存中即可，注意是“拷贝”，这是保证数据安全的手段，但也带来了很多问题，我们后文中会详细讨论。

这样的存储结构就是**外存**，长期存储程序和文件，备份主存数据。我们通常将外存分为两个层级，为的是节约成本和满足足够的运行效率：

##### 二级存储（Secondary Storage）

这是计算机最常见外存设备，核心作用是“在线存储”——直接为系统和应用提供可快速访问的长期存储，是主存的直接备份和扩展。

典型实例：硬盘（HDD，机械式存储）、固态硬盘（SSD，电气式非易失性存储）。

这一级中有一个特别的存储结构，EEPROM 固件，它专门用于开机时引导载入操作系统。

##### 三级存储（Tertiary Storage）

这类存储的特点是：容量大、成本低、速度慢，核心作用是“离线备份、长期归档”，不直接参与系统的日常运行，仅在需要备份数据、恢复历史数据时使用。

典型实例：磁盘、软盘、光盘（现在已经不常用，或者说整个三级储存都有些过时了）。

#### I/O控制器与设备控制器

负责 I/O 工作的硬件设备有很多（键盘、显示屏、鼠标、音响等），但它们的工作原理各异，且速度远慢于 CPU ，如何让与计算机一起工作呢？计算机使用了**I/O控制器（Controller）**（硬件）与这些设备直接交互。控制器主要有两个部分：

- 寄存器
- 局部缓冲区

I/O 设备将当前自身的状态存入控制器内的**寄存器**，将产生的交互信息存入其**局部缓冲区**。

随着计算机的发展，我们希望有更多专门设备（显卡，网卡等）能接入计算机，使其变得愈发强大。但一样的问题：它们工作原理各异，速度慢。所以仍然采用**设备控制器（Device Controller）**（硬件）与这些设备直接交互,结构一致。不难发现，I/O 控制器就是一种特殊的设备控制器。时至今日，多数电脑都使用集成了的设备控制器管理外部硬件。

处理器、存储、外接设备及其对应的设备管理器体现了完整的现代版的冯·诺依曼架构。

设备控制器、CPU、内存之间通过**系统总线**进行信息传输。

```
CPU ──┐
      ├──── 系统总线 ────┬── 设备控制器（磁盘）── 磁盘
主存 ──┘                 ├── 设备控制器（键盘）── 键盘
                         └── 设备控制器（网卡）── 网络
```

#### 机器码、例程与程序

计算机硬件只能直接执行**机器码（Machine Code）**，即由 0 和 1 组成的二进制指令序列。每一条指令都对应 CPU 内部的一个基本操作（如加法、数据移动、跳转等）。然而使用机器码编程不仅难以书写和阅读，而且易错。因此人们引入**汇编语言（Assembly Language）**，它使用助记符（如 ADD、MOV）来表示机器指令，并允许使用符号地址替代具体内存位置。汇编程序需要通过**汇编器（Assembler）**翻译为机器码后才能执行。

进一步地，随后人们发展出了更接近自然语言的**高级语言（如 C、C++、Java 等）**。这些语言不再关心具体的硬件细节，而是以逻辑结构（条件、循环、函数）来组织程序。高级语言程序需要通过**编译器（Compiler）**或**解释器（Interpreter）**转换为机器码或中间形式，才能被计算机执行。

典型的编译过程如下：

```
源代码（C/C++ 等）
        ↓ 编译（Compiler）
汇编代码（Assembly）
        ↓ 汇编（Assembler）
机器码（Machine Code）
        ↓
加载到内存 → CPU 执行
```

程序设计中人们会把常用的一段功能代码封装起来，形成可复用的代码段，这类代码称为**例程（Routine）**。多个例程按照一定逻辑组织在一起，构成一个完整的**程序（Program）**。

### 操作系统简史

#### 早期大型计算机

##### 单道程序——批处理系统

以 ENIAC、EDVAC 为代表的早期计算机运行方式是：程序员为特定任务进行专门编程，手动输入信息，但随着任务进一步复杂，常常出现 CPU 长时间等待 I/O：

```
CPU 处理信息：1 秒
等待 I/O 完成：30 秒  ← CPU 空转，浪费时间与能力
```

在 1950 年代，计算机处理器是由电子管连接而成，体积和价格都相当于一栋大楼。高昂的成本使得我们无法接受这样的空转浪费，我们迫切需要能提高计算机利用效率的方法，这就是人们发明操作系统的动机。

1956 年，General Motors 与 North American Aviation 联合开发了 GM-NAA I/O，运行于 IBM 704，这是第一个操作系统。它可以让计算机自动依次执行一批程序，不再需要在每个程序之间人工介入。

```
[作业 1] → [作业 2] → [作业 3] → ……
```

这就是**批处理系统（Batch System）**。它消除了人工操作的瓶颈，但 CPU 等待 I/O 的问题依然存在。

##### 多道程序系统

批处理系统中，一个程序等待 I/O 或者其他设备的输入时，CPU 仍然空转。自然的想法是：当程序 A 等待时，让 CPU 转去执行程序 B。这就是**多道程序设计（Multiprogramming）**的想法——让 CPU 在多个程序之间切换，不浪费等待的时间。1964 年 IBM 推出的 OS/360 实现了这一目标。

一个问题是：当 CPU 转去执行其他程序时，设备输入完成后如何通知它？

###### 中断机制

**设备中断**

第一种想法是让 CPU 持续检查设备的寄存器信息，这样的方式叫轮询，也叫**忙等待（Busy Waiting）**，但这样 CPU 会一直轮询，无法执行其他操作。

第二种想法是让 CPU 发出等待设备输入的指令后继续执行其他指令，设备控制器完成后主动向 CPU 发送信号，CPU 响应该信号，暂停当前执行流，转去处理设备事件，完成后恢复。这一机制称为**中断（Interrupt）**，更高效，于是被广泛采用。

中断机制的实现需要有硬件与软件程序共同参与：

在 CPU 上有一个引脚，设备控制器完成任务后拉高该引脚电平，通知 CPU 设备已完成操作，该引脚被称为**中断请求线（IRQ Line）** 。

现实中多个设备可能同时发出请求，因此主板上设有专用硬件芯片**中断控制器（Interrupt Controller，如 Intel APIC）**，收集各设备的中断请求，按预设优先级仲裁后向 CPU 发出信号并附带**中断编号**。中断控制器不存有任何程序，只负责信号的排队与转发。

CPU 收到中断编号后，用它索引操作系统在启动时写入主存低地址区的一张指针数组——**中断向量表（Interrupt Vector Table）**：

根据该地址，CPU 跳转至对应的**中断服务例程（Interrupt Service Routine，ISR）**——存放在主存中的一段程序代码，是实际处理中断事件的程序。跳转前，CPU 由其硬件机制自动将当前程序计数器与寄存器值压入栈；ISR 执行完毕后从栈中恢复，返回断点继续执行。状态的保存与恢复由 CPU 自身完成，与中断控制器无关。

```
设备控制器完成任务
  │  拉高 IRQ 线                              （硬件）
  ▼
中断控制器仲裁，向 CPU 发出中断编号            （硬件）
  ▼
CPU 执行完当前指令后检测到信号                 （硬件）
  │  保存寄存器状态至栈                        （硬件）
  │  以中断编号索引中断向量表，取 ISR 地址      （硬件读软件数据）
  ▼
ISR 执行：处理数据，通知驱动程序                （软件）
  ▼
CPU 从栈恢复寄存器状态，返回断点                （硬件）
```

中断按屏蔽性分为两类：

**不可屏蔽中断（NMI）**用于不可恢复的硬件故障（如内存错误、掉电），CPU 无论如何必须响应；

**可屏蔽中断（MI）** 用于普通设备 I/O 完成通知，CPU 可临时关闭响应。

这样区分的原因是确保一些关键操作不被打断，例如更新调度队列时，临时屏蔽可保证其完整性。

但是随着外设数量的增多，有时会出现设备数量超过中断编号的总容量，为了节省中断编号，现代计算机一般采取：让一些类似的设备使用同一个中断编号，但此时的向量表项都指向对应的同一个设备处理程序链表，轮询后在其中找到对应设备再跳到对应的例程，这样的方式称为**中断链（Interrupt Chaining）**。

**陷阱中断**

除了由外部设备发出的中断请求，操作系统还需处理另一类中断——**陷阱（Trap）**，这是由用户/系统软件触发的中断，分为两种情形：

**程序错误**，如除以零、访问非法内存地址。CPU 通过中断向量将控制权转交操作系统，操作系统通常终止该程序并输出错误信息。

**系统调用**(详见后)

但多道程序随即带来了新问题：

- 多个程序谁先执行？
- 如何分配内存？

1969 年，Bell Labs 的 Ken Thompson 与 Dennis Ritchie 设计的 Unix 操作系统通过建立统一的架构解决了这些问题。

同时，为了提高昂贵大型机的使用率，人们还希望计算机能支持多用户使用同一套硬件同时工作。 1960 年代中期，IBM 在大型机上开发了 CP-40 和 CP-67 实验系统，实现了这个功能——**虚拟化**，允许多个操作系统实例在同一台物理机上并发运行。

##### 多任务(分时)系统

多道程序解决了CPU 利用率，但依然是批量、非交互的，执行任务时用户不能操作。为了让用户在执行任务时还能使用计算机，即让用户感觉任务程序在“并发”地同时运行，**多任务(分时)系统（Time-Sharing System）** 应运而生。

1960s 中期的 CTSS（Compatible Time-Sharing System）是第一个成熟分时系统，该系统通过将各任务先分为不同的**优先级**，之后逐级执行；把 CPU 工作时间切分成极短的**时间片（time slice）**并在不同任务之间急速轮转切换等方式，制造了用户感官上的“并发交互”，且一台计算机被多个用户使用，因此每个用户都能感觉自己在独立的与计算机进行多任务交互。

#### 个人计算机

Unix 的成功奠定了现代操作系统的基石。20 世纪 70 年代末，随着处理器被逐渐集成变为一个部件—— **CPU（Central Processing Unit，中央处理器）**诞生，计算机具备了从机房走向个人桌面的条件。1977 年的 Apple II 和 1981 年的 IBM PC 第一次将计算机交到了普通人手中，这也催生了个人计算机操作系统的发展，时至今日，个人计算机上的操作系统已经成为系统中的绝对主流。

早期个人计算机的操作系统相对简单。1981 年，IBM 选择与微软合作，为 IBM PC 配备了 MS-DOS（Microsoft Disk Operating System）。MS-DOS 允许用户通过键盘输入指令与计算机交互，完成任务。

1984 年，苹果公司推出了 Macintosh，这个系统允许用户用鼠标点击屏幕中的图标与计算机交互，这种“所见即所得”的设计大大降低了入门门槛。微软紧随其后，在 Windows 95 中将这样的交互方式与 DOS 融合，成为当今个人计算机操作系统的经典范式。

除商业操作系统之外，自由与开源软件运动的兴起也对操作系统的发展起到了关键助推。1991 年，芬兰大学生 Linus Torvalds 出于学习目的，开发了一个类似 Unix 的操作系统，并将其命名为 **Linux**。开源后便吸引了众多开发者一同参与，系统得以逐渐完善。如今，Linux 是超级计算机、服务器和绝大多数手机设备底层操作系统。

#### 集群系统

单台计算机的能力受制于硬件物理，为了提升计算机的能力，人们自然想到：可以把多台计算机联合起来，这就是**集群系统（Clustered System）**。它将多台独立的计算机（称为**节点**）通过局域网或高速互联（如 InfiniBand）相连接，极大提高了计算机的能力上限。该系统还带来一个优点：当某个节点发生故障时，其他节点可立即接管其任务，而用户几乎毫无察觉，提升了任务的稳定性。

#### 分布式系统

更广义地，将更大更多在地理上分散的计算机通过网络连接、共同对外提供服务的系统，称为**分布式系统（Distributed System）**。

#### 实时系统

还有一类操作系统，它们的核心目标不是“求上限”——尽可能提高效率或是提升用户体验，而是“保下线”——必须在规定的时间内组织硬件完成任务，一旦错过截止时间，就可能造成严重后果，这就是**实时系统（Real-Time System）**。它广泛应用于汽车发动机控制、工业机器人、医疗成像设备、航空航天等场景。

以上是对计算机操作系统发展的一个梳理，引入的相关概念和系统的具体组织将在后文介绍。本讲义将主要聚焦于现代个人计算机的操作系统的组织方式，需要特别说明，即使仅讨论这部分，目前的情况是各系统之间的组织方式仍然和而不同，各有所长。我们的介绍策略是先了解共性，再对比不同。对于共性部分，我们将回到现场，站在当时的架构者们身旁，感同身受地了解当时的困境，做出取舍；对于不同点，我们将了解他们各自的需求，再横向对比，论短说长。话不宜迟，雄关漫道真如铁，而今迈步从头越，让我们开始吧。

#### 进程与线程

##### 进程

我们把运行中的程序称为**进程（Process）**。

从程序变为进程需要经过两个关键步骤：

- **链接（Linking）** 一个程序通常由多个源文件编译而成，每个源文件会生成独立的目标文件（Object File）。此时就需要将这些目标文件以及所需的库文件合并，解析其中的符号引用（如函数调用、全局变量），最终生成一个完整的可执行文件（Executable），这就需要**链接器（Linker）**
- **加载（Loading）** 程序只有在内存中才能被运行，因此我们需要将可执行文件从外存读入内存，这就需要**加载器（Loader）**，加载器还会完成以下工作：
- 分配进程所需的内存空间
- 将程序代码和数据映射到内存
- 初始化程序运行所需的上下文（如栈、寄存器）
- 设置程序入口地址（即 main 函数）

##### 线程

我们从什么时候有了区分程序是否正在运行的需求的？回想一下，假如面对没有操作系统，或仅装有批处理系统的计算机，根本没有必要区分——你就算知道了也不能改变什么。所以只有操作系统能“切换程序避免浪费等待设备 ”后，我们才有区分的必要。

对于多任务操作系统——能切换以提供程序同时运行的假象，也叫**并发**，我们通过频繁切换进程也可以实现。但是，这些早期的操作系统和今天你手边的计算机里的操作系统最大的区别是什么？——面向的用户数量不同，从公用计算机到个人计算机的转变带来一个问题：“并发”标准改变了。

公用计算机时代，并发的目标是：每个用户之间的公平性和整体硬件的利用率。每个用户只运行一个进程，他们唯一的要求是：“我别等太久”

但在个人计算机时代，并发的目标变为：单个进程的响应性和多开进程的并行需求。用户需要的是一次运行很多个进程，而且要求每个进程不能太慢，这里的关键点是：“我需要同时做很多事”。

在个人计算机的兴起、并行定义的改变、图形化的流行、程序自身的复杂化等背景下，如果还仅使用“进程”来实现”并发“不但在技术上十分困难 ，在学术上描述计算机的行为上也不够清晰。于是，后续编写的程序采取了将最终任务拆分为多份的方法，每一份的“小任务”都由一个“小进程”负责，它们共享数据，但又独立推进。最后再把这些“小进程”的结果合并，完成任务。我们现在把这些“小进程”称为一个进程的**线程（Thread）**。

### 操作系统的组成部分

历史上，早期的操作系统并没有清晰的结构划分，批处理系统乃至早期多道程序系统中，操作系统往往是“铁板一块”的一个程序：所有功能都直接写在一起，彼此交织，形成一个紧密耦合的整体。

但随着计算机能力的提升和需求的增加，操作系统的功能迅速膨胀，问题开始显现：

- 难以维护：修改一个功能可能影响整个系统
- 难以扩展：新增功能必须深入已有代码内部
- 难以验证：系统行为难以分析与推理

为解决这些问题，人们逐渐形成了一种设计思想，即对系统本身进行“拆分”，分块编写与维护。在操作系统中，这种思想的应用方式是：

- 将稳定、通用、必须正确的部分的实现抽取出来作为里层
- 将可变、依赖需求的部分的实现留在外层，允许灵活调整

于是操作系统逐渐形成了清晰的结构分层：

- 里层：集中实现基础能力，例如与硬件交互、常用的基础功能实现
- 外层：基于里层对接用户，根据用户需求进行

可以发现在这样的设计里，系统的里层是外层的基础，用户只与外层交互，外层和里层交互，里层负责实现。对于普通用户使用时并无需功能的实现原理，他们只需通过内层将自己的请求发送到里层，这样的请求称为**系统调用 （System call）**。需要注意的是，系统调用并不是用户的自然语言式的要求，而是提前由架构师写入内核的一些“能力词条”——**系统接口**，用户只能请求接口内的系统调用。

#### 内核

我们把系统的里层部分称为**内核（Kernel）**，是系统中重要的程序集合，是最核心组成部分。内核开机即刻载入主存执行。

一个自然的问题是：到底哪些程序算“重要”，哪些算是“基础功能”？换句话说，哪些功能才算“内核”呢？事实上，这个问题的答案全凭架构者的心意，这里给出一个常见的分类，这是一个由历史演化的路径依赖和计算机硬件结构共同作用得到的一个收敛结构。

之前说过现代计算机的硬件有：CPU、内存(包括储存)、各种设备，所以内核应有对应管理之责：

- CPU → 进程管理（管理CPU的时间和资源分配，决定程序运行顺序）
- 内存 → 内存管理（管理内存分配）
- 内存和储存之间的数据交互 → 文件管理（管理存储设备上的数据组织与保存）
- 设备 → 设备管理（管理设备）

随着系统自身的发展，历史上为解决新问题，内核增加了如下对应管理责任：

- 批处理系统中 CPU 等待 I/O 空转 → 中断管理（让 CPU 被任务驱动，而非空转等待）
- 多道程序中多个程序同时驻留内存 → 同步管理（协调并发对共享资源的访问）
- 多任务系统中用户需要交互式响应 → 进程管理（合理调度对应程序执行）
- 多用户/多程序需要隔离与安全 → 接口管理（提供内核入口）
- 大型机需要在一台机器上运行多个操作系统 → 虚拟化管理（隔离与复用物理硬件）

##### 设备驱动

内核中时刻运行着**设备驱动（Device Driver）**， 该程序负责与设备控制器通信，发送指令、读取信息、处理硬件细节。

操作系统收发的信息都遵守固定的格式——接口。于是，对于操作系统的上层应用，有了统一的接口信息使应用程序不必关心硬件差异； 对于下层的设备驱动则需充当”翻译“，把接口信息转换为硬件能执行指令操作。

### 外层系统

我们把除内核外的其余操作系统的功能实现称为外层系统

#### 用户接口

**用户接口（User Interface, UI）**是人机交互的入口。主流的有以下几种：

- **命令行界面（CLI）**

可以读取用户输入的文字命令。

- **图形用户界面（GUI）**

将信息可视化，通过鼠标点击或拖拽交互。

- **触摸屏界面**

通过手势（点击、滑动）操作，屏幕上会模拟键盘进行交互

三种界面并非优劣之分，而是使用场景与用户类型的匹配：

| 界面类型 | 典型用户           | 核心优势                                   |
| -------- | ------------------ | ------------------------------------------ |
| CLI      | 系统管理员、开发者 | 高效、可编程（Shell Script）、完整功能覆盖 |
| GUI      | 普通用户           | 直觉友好、学习曲线低                       |
| 触摸屏   | 移动设备用户       | 自然交互、无需外设                         |

不论通过哪一种形式与用户交互，最后都会得到用户交互后产生的信息，这些信息并不能直接被操作系统理解继而输出信息。因此，我们需要一个”翻译“把交互信息译为操作系统能理解的系统调用指令。这就是**命令解释器（Command Interpreters）**，最常见的如 shell。首先它将 UI 得到的数据翻译为系统调用指令交给操作系统，操作系统调用硬件操作后得到信息返回给命令解释器，命令解释器再将数据”翻译“为对应的交互数据呈现给用户。

值得注意的是：命令解释器在 CLI，GUI或是触摸屏界面都存在，但是后两者的情况里可能会被整合为一个进程下的不同线程。macOS 则是一个有趣的”混血”：其底层基于 UNIX 内核（Darwin），上层覆盖 Aqua 图形界面，同时保留了完整的命令行入口。这使得 macOS 成为少数同时满足开发者与普通用户需求的系统之一。

### 从启动到运行：操作系统的视角

当计算机通电以后，遵循以下过程完成开机：

```
             上电
              ↓
       执行加电自检（POST）,使用 EEPROM 固件 (常见如 BIOS)  
              ↓  将必要的操作系统部件从 HDD/SSD 复制至主存
            [主存]  内核开始运行
              ↓
          计算机就绪
```

电源接通，CPU 的复位电路将其指向 EEPROM 中一个固定的地址——存放着固件（如 BIOS）代码。固件首先执行加电自检（POST），确保基本硬件正常工作。

然后，它根据配置的启动顺序，从第一个启动设备（某个二级储存）的特定地址开始读取引导加载程序。

引导加载程序的任务是将操作系统内核从磁盘加载到内存中，在加载之前，它需要做一些准备工作，比如切换 CPU 到保护模式或长模式，建立初始的内存映射。如果系统中有多个操作系统，引导加载程序会让用户选择。

一旦内核被加载到内存并获得 CPU 控制权，它首先会检查硬件，建立关键的数据结构（如进程表、内存管理表），初始化中断处理程序，并设置系统时钟。随后，内核会启动第一个用户级进程——在 Unix/Linux 系统中通常是 init 进程（在现代 Linux 发行版中可能是 systemd），在 Windows 中是 smss.exe（会话管理器）。

第一个用户进程负责启动更高层次的服务。例如，它会启动负责登录的服务（如 getty 或登录管理器），初始化网络服务，并最终启动图形界面（如 X Window 或 Windows 桌面）或命令行解释器（Shell）。

此时，计算机才真正准备就绪，静静等待用户的指令。而用户每发出一次指令，背后又是无数次操作系统服务的调用——打开文件、创建进程、分配内存、读写磁盘……操作系统这位“管家”，始终在幕后不知疲倦的协调着一切。
